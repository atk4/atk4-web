Object Oriented Web Programming

OOP is one of the major concepts in the world of Computing. For decades Object Oriented Programming has been used in all environments.

Web Programming, however, was initially design-driven. Scripting was supposed only to fill the gaps in static HTML files.

Agile Toolkit introduces the "Real" object-oriented web development. The principle of "abstraction" and "black boxes" is applied on all parts of web development - Models, Views, Controllers, Pages and Applications itself.

What are Web Objects?

$page->add('Button')->set('This is object');

Observe the button on the right. To your browser it's a sequence of HTML code. However for a developer in Agile Toolkit, the button is the object. The developer may not want to know how the button presents itself, how it reacts to events or how it's communicating with the database. Like many other things around it, the button is just another object.

Without thinking too much about "how?" a developer can now focus on "what?". Need More info about Object-Oriented Programming?

Objects are part of our real life. Take a car, for example. A car is an object. It has properties, such as color, model, year. There are things you can do with a car: drive, wash, lock and unlock it.

The goal of Object Oriented Programming is to discover self-sufficient objects in the software, and describe them using classes. The behaviour and the properties of the button are described in the Button class.

Properly constructed Object-oriented architecture hides the complexity of things, while still allowing a developer to be in control of the behaviour. With the great Object-oriented capabilities introduced to PHP since PHP5, web applications based on PHP can also be very object-oriented.

Agile Toolkit implements a general object-oriented architecture, as well as large amount of base classes. You can use the classes you like, and build upon them. Programming with Agile Toolkit is like building a sky-scraper: you build upon the foundation. You re-use, extend, enhance the basic building blocks.

The Object-Oriented Universe of Agile Toolkit

When Object Oriented Design is applied to Web Development, it abstracts the way you operate with Web Objects such as forms, buttons, pages, links and others. Instead of writing HTML, CSS ad JavaScript code, you now interact with a native object in PHP.

The button example demonstrates just one object in Agile Toolkit. With more complex objects we can implement forms, signup wizards, payment forms, file managers, polls, comment feeds and so on. Regardless of which object you are using, it's straightforward and simple.

When developing with Agile Toolkit, you will design your own objects, too. Primarily you will need to deal with two types of objects: Models and Pages. However, there are five major object types in Agile Toolkit.

Fives types of objects in Agile Toolkit

I have only mentioned "Visual" objects, or views. There are actually 5 object types in Agile Toolkit, and they are described below.

Application (API)

In simple terms, an Application (web application) helps the user manage server-side data. For each request made by the user, the web application needs to collect information from the data storage, and produce a response to the user.

Pages

A Page is a page. That's what we see when we open a book. It's also what we see when we open a website. On a website we navigate between the pages using hyperlinks. The pages in a website can look very different from each other, but they can also share many common components, such as a navigation menu, the general layout of the site, or the general 'look' (style) of the site.

It makes sense to build a page using an object-oriented approach. This way one page may easily 'inherit' another. For example: you might have a generic 'signup' page, and easily develop a 'registration' page which will 'inherit' the functionality of the signup, and extend it - like in our example.

Models

Models are objects mimicking real-world objects. Depending on your application, you will define different types of models. For instance: for a job site, you may have JobApplication, User and other similar objects.

Again, the object-oriented approach makes a lot of sense for modelling your application classes. If you have different user types, they would extend the same parent class.

Some plug-ins may include some generic models. For instance there is a model for "Country", along with a database table with content for the model. It contains the official list of ISO countries and codes. Including such a model can save you development time and help you update your application when the country list is updated or changed.

Views

Views are Widgets. While widgets in many frameworks are programmed using procedures and functions, the Views in Agile Toolkit are also implemented using Object-Oriented design: as classes. One View can inherit another View, or it can contain other views within itself.

Procedural widgets are aimed at end-users. You, as developer, can understand the benefit of an object-oriented approach better. That's why views in Agile Toolkit are not designed to serve all purposes, instead they are designed to be simply extended and tweaked, to work the way you need.

Controllers

Your application needs a lot of functionality. Some of it will help you deal with specific problems, such as template parsing, or database access.

Controllers in Agile Toolkit enhance functionality of their parent object. Controller may enhance the way errors are displayed, the way audit is done on your models, or how pagination works in your grid.

As controllers often have common functionality, object-oriented architecture can help us out here. For instance: you could have different authorization mechanisms - different authorization controllers - based on a generic authorization class.

What is Abstraction?

The Button object is designed to do one thing only: implement a button. The Form object, on the other hand, is meant to implement forms. It may, however, rely on a Button in order to do its job.

Going further up, a Page object's job is to implement a single page. The Page relies on objects such as Forms, Buttons blocks of text and so on. Finally the top-most object - which relies on Pages, Authentication, Database connectivity, many other objects, and which is meant to implement your whole web application - is the Application object.

All of the objects have a few qualities in common. They rely on each other to do their job. They hide the complexity of their implementation behind their interfaces.

As a good and efficient developer, your job is to understand the importance of abstraction, and to be able to implement additional objects to fit in their proper roles. Objects must always take advantage of other objects if possible, and each object must be simple to use.

Moving forward

To learn Agile Toolkit, you need to first understand the concept of Applications and PMVC (Page, Model, View, Controller). You don't have to understand in detail how each of these objects work right away, but as you learn how each one works, and how they fit in with each other, you'll gradually understand and appreciate the structure of the framework as a whole.

The approach we take in the Learning chapter will focus on one thing at a time. However, the first thing is to get your own copy of Agile Toolkit, so you can try things out. If you do not wish to install Agile Toolkit yet, you can skip the installation part.
